# Учёт расходов — веб-приложение команды

Русскоязычное приложение для совместного учёта расходов. Node.js + Express +
PostgreSQL. Без Supabase, без Telegram, без постоянно работающего сервера
где-то ещё — всё работает на Railway. Авторизация без паролей: выбор имени из
заранее заданного списка, выбор запоминается на устройстве на 1 год.

## Структура проекта

```
expense-tracker/
├── server/              # Backend (Express, работает только на сервере)
│   ├── index.js         # Точка входа
│   ├── db.js            # Подключение к PostgreSQL (только на сервере!)
│   ├── auth.js          # Определение текущего пользователя по X-User-Id
│   ├── migrate.js        # Применение SQL-скриптов из sql/
│   ├── expenseQueries.js
│   ├── csv.js
│   └── routes/
│       ├── users.js
│       ├── categories.js
│       ├── expenses.js
│       └── reports.js
├── public/              # Frontend (статика, отдаётся тем же сервером)
│   ├── index.html
│   ├── css/style.css
│   ├── js/app.js
│   ├── manifest.json     # PWA
│   ├── service-worker.js
│   └── icons/
├── sql/                 # SQL-скрипты для PostgreSQL
│   ├── 001_create_tables.sql
│   ├── 002_seed_categories.sql
│   ├── 003_seed_users.sql
│   ├── 004_indexes.sql
│   └── 999_migration_template.sql
├── .env.example
├── package.json
├── railway.toml
└── Procfile
```

Браузер пользователя никогда не подключается к PostgreSQL напрямую — только
к Express-серверу по HTTP(S). Строка подключения `DATABASE_URL` существует
только как переменная окружения на Railway и никогда не попадает в
репозиторий или во фронтенд-код.

## Как это устроено

- **Вход без пароля.** При первом заходе показывается список имён
  (`GET /api/users`). Выбор сохраняется в `localStorage` и cookie на 1 год.
  Дальше при каждом запросе, который должен быть привязан к человеку
  (добавление расхода, админ-действия), фронтенд посылает заголовок
  `X-User-Id`, а сервер проверяет пользователя в базе.
- **Роли.** У пользователя есть `is_admin`. Администратор (по умолчанию —
  Роман) видит в «Профиле» блок управления пользователями и категориями.
- **Расходы неизменяемы в v1.** Осознанное решение по ТЗ: чтобы не потерять
  и не испортить историю, редактирования и удаления расходов нет. Ошибку
  исправляют новой записью с пояснением в комментарии.
- **Отчёты и экспорт.** Список операций, сводка по категориям (с группировкой
  по разделам) и сводка по пользователям. Экспорт — CSV с BOM (корректно
  открывается в Excel на Windows).

## 1. Установка и запуск локально

Понадобится Node.js 18+ и доступ к любой PostgreSQL-базе (можно временную,
например через Railway, Docker или локальный Postgres).

```bash
cd expense-tracker
npm install
cp .env.example .env
# впишите в .env свою строку подключения DATABASE_URL
npm run migrate   # создаст таблицы, категории и начальных пользователей
npm start          # http://localhost:3000
```

> Скрипты в `sql/` и `server/migrate.js` идемпотентны — их можно запускать
> повторно без риска что-то сломать или задвоить данные.

## 2. Как создать GitHub-репозиторий и загрузить проект

1. Зайдите на [github.com](https://github.com) → **New repository**.
2. Укажите имя, например `expense-tracker`, оставьте репозиторий **Private**
   (в нём нет секретов, но это внутренний рабочий инструмент команды).
   Не добавляйте `.gitignore`/README при создании — они уже есть в проекте.
3. На своём компьютере, в папке с проектом:
   ```bash
   git init
   git add .
   git commit -m "Первая версия учёта расходов"
   git branch -M main
   git remote add origin https://github.com/<ваш-логин>/expense-tracker.git
   git push -u origin main
   ```
4. Проверьте на GitHub, что папки `server/`, `public/`, `sql/` и файл
   `.env.example` загрузились, а файла `.env` там нет (он в `.gitignore`).

## 3. Как создать проект в Railway и подключить GitHub

1. Зайдите на [railway.app](https://railway.app) → **New Project**.
2. Выберите **Deploy from GitHub repo**, авторизуйте Railway к своему GitHub
   (если ещё не подключали), выберите репозиторий `expense-tracker`.
3. Railway определит Node.js-проект автоматически (Nixpacks) и начнёт
   собирать `web`-сервис. Не переживайте, если первый деплой упадёт — база
   ещё не подключена, это нормально на этом шаге.
4. Каждый `git push` в ветку `main` теперь будет автоматически запускать
   новый деплой на Railway.

## 4. Как создать PostgreSQL-базу в Railway

1. В том же проекте Railway нажмите **+ New** → **Database** → **PostgreSQL**.
2. Railway создаст сервис Postgres и сам сгенерирует переменные подключения
   (`PGHOST`, `PGPORT`, `PGUSER`, `PGPASSWORD`, `DATABASE_URL` и т.д.) —
   они видны на вкладке **Variables** сервиса Postgres.

## 5. Как добавить переменную DATABASE_URL в сервис приложения

1. Откройте сервис приложения (не Postgres) → вкладка **Variables**.
2. Нажмите **New Variable** → **Add Reference** (или аналогичную опцию) и
   выберите `DATABASE_URL` из сервиса Postgres — так значение всегда будет
   актуальным, даже если Railway его поменяет.
   Если такой опции нет в интерфейсе — просто скопируйте значение
   `DATABASE_URL` со страницы Postgres и вставьте его как обычную
   переменную в сервисе приложения.
3. При необходимости добавьте `PGSSL=false` (обычно не требуется — значение
   по умолчанию). `PORT` задавать не нужно — Railway подставляет его сам.
4. Сохраните — Railway передеплоит сервис с новыми переменными.

## 6. Как запустить SQL-скрипты (создание таблиц, категорий, пользователей)

Это происходит **автоматически** при каждом деплое: команда запуска в
`railway.toml` — `npm run migrate && npm start`. При первом успешном деплое
после подключения базы таблицы, разделы, категории и пользователи (Роман —
администратор, а также Влад, Саша, Анна, Александр) создадутся сами.

Если нужно запустить миграцию вручную (например, чтобы проверить логи):
1. Откройте сервис приложения в Railway → вкладка **Deployments** → откройте
   активный деплой → или используйте кнопку **Shell** сервиса, если доступна.
2. Выполните: `npm run migrate`.

## 7. Как добавить нового пользователя

**Через приложение (проще всего):** зайдите под Романом → «Профиль» →
«Пользователи» → «+ Добавить пользователя» → введите имя.

**Через SQL (Railway → сервис Postgres → вкладка Query/Data):**
```sql
INSERT INTO users (name, is_admin) VALUES ('Имя', FALSE)
ON CONFLICT (name) DO NOTHING;
```

## 8. Как изменить список категорий

**Через приложение:** «Профиль» → «Категории» → значок ✎ переименовывает,
значок ✕ удаляет (если по категории уже были расходы — она не удалится
физически, а просто скроется из выбора, чтобы не потерять историю).
Добавление — кнопка «+ Добавить категорию».

**Через SQL:**
```sql
-- переименовать
UPDATE categories SET name = 'Новое название' WHERE name = 'Старое название';

-- добавить категорию в существующий раздел
INSERT INTO categories (name, group_id, sort_order)
SELECT 'Новая категория', id, 99 FROM category_groups WHERE name = 'Прочее'
ON CONFLICT (name, group_id) DO NOTHING;

-- скрыть категорию (не потеряв историю)
UPDATE categories SET is_active = FALSE WHERE name = 'Категория';
```
См. также `sql/999_migration_template.sql` — шаблон для будущих изменений
структуры базы.

## 9. Как подключить свой домен к Railway

1. В сервисе приложения → вкладка **Settings** → раздел **Networking** →
   **Custom Domain** → **+ Add Domain**, укажите свой домен/поддомен.
2. Railway покажет CNAME-запись — добавьте её у своего регистратора домена
   (или в DNS-панели, например Cloudflare).
3. Дождитесь выпуска SSL-сертификата (обычно несколько минут, иногда дольше)
   — после этого домен откроется по HTTPS.

## 10. Резервное копирование и экспорт в CSV/Excel

**Экспорт расходов в CSV (для Excel):** в разделе «Отчёты» выберите нужный
вид отчёта и период, нажмите «Экспорт в CSV» — файл скачается с кодировкой
UTF-8 + BOM, в Excel на Windows кириллица отобразится корректно.

**Полный бэкап базы данных (pg_dump):**
```bash
# DATABASE_PUBLIC_URL — берите из вкладки Variables сервиса Postgres в Railway
# (используется публичный хост, а не внутренний, чтобы подключиться со своего компьютера)
pg_dump "$DATABASE_PUBLIC_URL" -F c -f backup-$(date +%Y-%m-%d).dump
```
Также можно настроить регулярный бэкап через **Railway → сервис Postgres →
Settings → Backups** (если доступно на вашем тарифе).

## 11. Как восстановить базу из резервной копии

```bash
pg_restore --clean --if-exists -d "$DATABASE_PUBLIC_URL" backup-2026-07-23.dump
```
⚠️ `--clean` удалит текущие данные перед восстановлением — используйте на
пустой/тестовой базе или когда уверены, что это нужно.

## 12. Как перенести проект, базу и домен на другой Railway-аккаунт

1. **Код:** он уже в GitHub — просто подключите тот же репозиторий к новому
   Railway-аккаунту (шаг 3 выше).
2. **База:** создайте новый сервис PostgreSQL в новом аккаунте (шаг 4), затем
   перенесите данные:
   ```bash
   pg_dump "$OLD_DATABASE_PUBLIC_URL" -F c -f transfer.dump
   pg_restore --clean --if-exists -d "$NEW_DATABASE_PUBLIC_URL" transfer.dump
   ```
   Либо просто выполните SQL-скрипты из `sql/` на новой базе (если старые
   данные переносить не нужно, а достаточно структуры и стартовых данных).
3. **Переменные окружения:** пропишите `DATABASE_URL` нового Postgres в
   сервисе приложения (шаг 5).
4. **Домен:** отвяжите домен от старого проекта (Settings → Networking →
   удалить Custom Domain) и добавьте его в новом проекте (шаг 9), обновите
   CNAME-запись, если Railway выдаст новое значение.
5. Всё это делается без участия Claude — только GitHub и панель Railway.

## 13. Внешние сервисы, от которых зависит приложение

- **GitHub** — хранение исходного кода и SQL-скриптов, триггер автодеплоя.
- **Railway** — хостинг Node.js-сервера, база данных PostgreSQL, домен.

Больше ничего: ни Supabase, ни Telegram, ни сторонних API. Доступ к
приложению — по прямой ссылке на Railway-домен (или ваш собственный домен),
без публикации и без индексации.
